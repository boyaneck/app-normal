# AI Copilot 서비스 설계서 (Technical Specification)

| 버전  | 날짜       | 수정 내용                      |
| :---- | :--------- | :----------------------------- |
| v 1.0 | 2026-05-14 | 초안 작성 (핵심 시나리오 정의) |

이 문서는 스트리밍 호스트의 방송 품질 향상 및 수익 극대화를 위한 **AI Copilot** 기능의 상세 기획 및 데이터 구조를 정의.

---

## 1. 개요 (Overview)

AI Copilot은 방송 중 발생하는 다양한 데이터를 실시간으로 모니터링하고, **음성 인식(STT)**을 통해 호스트와 소통하며 방송 지표를 관리하는 지능형 대시보드 서비스.

---

AI Copilot 엔진이 실시간 분석 및 트리거를 발생시키기 위해 사용하는 핵심 데이터 구조.

## 2. 기술적 구현 방향 (Implementation)

- **Speech-to-Text**: Web Speech API를 활용한 실시간 음성 명령 인식.
- **Real-time Logic**: **Redis Pub/Sub** 및 **Socket.io**를 활용한 지연 없는 데이터 전달.
- **UI/UX**: **Glassmorphism** 테마를 적용하여 방송 화면 위에 자연스럽게 녹아드는 사이드 패널 구성.

## 3.아키텍쳐

[음성입력] - STT 서비스( open AI Whisper 사용,한국여 처리 잘됨 )

- 의도 분류하기, 음성이 텍스트로 변환되면서 유저의 물음이 "조건을 설정하는것인지, 현재 지표를 묻는것인지"를 의도를 분류하기
- local server에서 유저의 의도에 따른 질문을 라우팅 처리하고 그에 맞는 프롬프트 작성하기 (LLM)

데이터 파이프라인

- 기존의 실시간(Redis) 수집 중인 데이터 이용
- 조건 비교시 단순 수치 비교는 "AI"질문 하기 전 로컬 서버가 먼저 빠르게 처리
- 복잡한 조건 (현재 상황 파악, 복잡한 조건만 LLM 처리 )
- 처리된 AI 답변을 유저에게 보여주며 UI 처리

## Agent 설계

- 키워드를 기반으로 필요한 데이터를 추출하여 LLM 호출(function calling은 규모가 커졌을 때 적용하도록 )
- 총 2번 호출하여 LLM(AI)이 질문에 알맞는 데이터를 선택하는 구조로 파이프 라인 설계
- DB에서 직접 조회한 데이터만 응답
- 새 기능 추가 시 함수만 추가,프롬프트 변경을 최소화 하는 방식
- 모든질문에 Agent 패턴이 적용되지만,가능한 빈번하고 Agent를 거치지 않아도 되는 질문이 있다면 분리여 로직처리(가벼운 LLM 호출 지양)

## 데이터 처리

- EMA (지수이동평균)을 적용하고 이 가중치를 통해 copilot의 인지 상태를 더 세밀하게 향상
